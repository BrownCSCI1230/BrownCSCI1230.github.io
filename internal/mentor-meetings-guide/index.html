Mentor Meetings Guide

As a CS 1230 TA, you will be meeting with your assigned students on a weekly basis in order to gauge their progress in the course and understanding of the material through project grading and, optionally, lab checkoffs.

Before your meetings

Find the submission commit id

On Sunday after 10:30 AM ET (72 hours after the due date), find the row corresponding to each student in the Student Information spreadsheet.

A link to their GitHub repository should automatically be populated, and the latest commit before the late deadline should be populated by a script. If there is an error, then try to find the student's repository manually, and look for the latest commit before the late due date.

If a student has an accommodation to submit an assignment more than three days late, see the section on Late Submissions.

Email your students

Copy and fill out the following email template and message your students on the Sunday after each project deadline. If the student has not submitted or has extenuating circumstances which have resulted in a need to submit more than 3 days late, do not send them this message but instead do one of the following:

  1. Schedule a one-off meeting after they finish
  2. Ask an HTA to schedule a one-off meeting with the student
  3. Cancel this week’s meeting and make a plan to fit this week’s material into next week.

Initial grading

Overview

Before your block of mentor meetings and after the late deadline, you will do an initial quick grading pass on each student’s submission. In this initial pass, you will give points as conservatively as possible, only for output which is clearly correct. For cases which warrant partial credit, make a note to discuss with the student during your meeting.

For algo submissions, you will simply check the student's final answer and make sure a reasonable amount of work is shown. If the student has a correct answer but little work shown, make sure to ask the student to explain their thought process during the meeting.

For code submissions, this initial pass consists of comparing expected output and student output in the corresponding section of the README.md file at the repository root. If the student has not filled out their README.md and does not have an accomodation to submit more than 3 days late, email them and remind them to fill it out. In addition to comparing outputs to evaluate correctness of functionality, you should briefly evaluate the submission's code quality and style according to the following general principles:

  1. Organization: Is there at least some degree of organization, such as separating code into multiple files or functions?
  2. Efficiency: Is the code reasonably efficient? For example, are there any obvious inefficiencies such as unnecessary loops, redundant calculations, or unnecessary memory allocations or copying?
  3. Readability: Is the code reasonably readable? For example, are there any obvious readability issues such as overly long functions, overly long lines, or overly complex logic?

Opening the project

In the student Mentor Meeting guide, students are given the following instructions for opening their project at the commit id listed in the Student Information spreadsheet (and which you will have emailed to them):

  1. Open a terminal and navigate to the project directory.
  2. Run git status to make sure you are on the correct branch and have no uncommitted changes. If you have uncommitted changes, commit them or stash them with git stash. You can unstash them later with git stash pop.
  3. Run git checkout <commit id>. This will put you in a detached head state, which is fine.
  4. Run git status again to make sure you are on the correct commit.
  5. Run qtcreator . to open the project in Qt Creator. If you do not have the qtcreator command installed, you can open the project by opening Qt Creator and selecting File > Open File or Project... and selecting the project directory.
