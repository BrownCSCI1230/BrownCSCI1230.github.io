Project 5: Logan Give Us A Name

Introduction

In the Ray assignments, you implemented a ray tracer that projects a 3-dimensional scene on a 2-dimensional plane. Ray-tracing, as you probably have experienced, can be very slow.

In this assignment, you will design a real-time sceneviewer. In other words, you will implement from scratch the Real-Time Graphics pipeline using components from previous labs including Parsing, Transformations, Trimeshes, VAOs, and Shaders.

Deliverables

  • Tesselation for the following shapes
    • From Trimeshes Lab
      • Sphere
      • Cube
    • Implement for this project
      • Cone
      • Cylinder
  • Port your SceneParser from lab 5
  • Camera
    • Extend implementation with perspective projection
  • Setup RealTime pipeline to render scene from parsed metadata

    • pass in shader uniforms

The Stencil

You may notice the stencil code provided is minimal, that is by design. To complete this assignment you will need to have a good understanding of the OpenGL pipeline.

We have provided for you the following. - GlRenderer : An OpenGL context to draw - Settings : Initialized with the same main.ini as Ray

And you have already written the following. - A scene parser (Lab 5) - A basic camera class (Ray Project)

Trimeshes Part 2

In order to render more complex scenes, you are going to need to implement a tesselator for two new shapes: a Cone and a Cylinder.

We recommend you implement these shapes in the trimeshes lab code and then copy over the relevant generation code. This will allow you to use the visualizer to debug position and normals.

While the specifics of your tessellation code are up to you, you are expected to design your program in an extensible, object-oriented way. This means minimal code duplication and no 1000 line branch structures. You will lose points if you do not follow these guidelines.

Detailed Specification

The cone and cylinder should have the following properties...

  • height 1
  • base diameter 1
  • axis aligned vertically along (0,1,0)
  • the object space origin is centered halfway (0.5) along the central vertical axis.

The tesselation parameters should control the vertical and radial tesselations like they did for the sphere.

Be especially careful when calculating normals for the tip of the cone, make sure that they are aligned with the surface of the cone and not pointing straight up.

SceneParser

As we are using the same scenefile format from Ray for Realtime, you will need to port over your scene parser to convert the raw scene data into renderable objects.

Of course the final functionality of the scene parser will depend on how you design your classes and abstractions.

Camera design

Your camera for Ray only needed to produce a view matrix. For Realtime, you must produce both a view and projection matrix given the scene file's camera parameters. The Projection matrix is needed to convert from camera space to clip space for OpenGL to render the scene correctly.

To implement your view matrix, you are not allowed to use glm::lookAt.

To implement your projection matrix, you are allowed to use glm::perspective

You will need to design your camera in an extensible manner as we will be expanding on this funcitonality in Realtime 2.

Realtime Pipeline

Welcome to the Meat and Potatoes of the project. You will use everything you have learned from lecture and the labs and manipulate the OpenGL pipeline to render the scene data.

You will take your parsed scene metadata, use it to construct all necessary VAO/VBO objects and all shader uniform data. Then you will use these in the main render loop of paintGL to finally render the scene.

To enforce good program design and to make you think about the design behind the functionality of OpenGL, we will only allow you to directly call a small amount of gl* functions within initializeGl(), paintGl() and resizeGl() the rest must be contained within abstract and extensibly designed classes.

You are allowed to directly call...

  • glClearColor
  • glEnable
  • glClear
  • glDisable

All other OpenGL functions may not be called directly.

Shader Program functionality

As you have implemented a basic phong model in the shader lab, we will ask you to extend that shader with support for object and light colors.

Thus your shader should, fow now, have the following features...

  • directional light support only
  • ambient, diffuse, and specular intensity computation
  • final color computation integrating both object and light color

Of course your shader must also perform all the necessary coordinate space conversions (think MVP matricies) as well as the illumination model computation.

Extra Credit

Remember that half-credit requirements count as extra credit if you are not enrolled in the half-credit course.

  • Adaptive level of detail Going a step further, you can implement varying degrees of tessellation depending on how many objects in the scene there are. In other words, for scenes with many objects in them, each object should be rendered with fewer triangles than if the scene contained a small number of objects. Look up level of detail for more information. Note: many of the examples you will find discuss distance-based or view-dependent level of detail, which you are not required to do!

  • Create your own scene file Create your own scene by writing your own scene file. Refer to the provided scenefiles for formatting. Your scene should be interesting enough to be considered as extra credit (in other words, placing two cubes on top of each other is not interesting enough, but building a snowman with a face would be interesting!).

  • Mesh Loader Write your own script to load in .obj and load it into your scene. We would need to provide some scenes that make use of this feature if we are going to include it as extra credit.

  • Torus Add in tesselation support for a torus.
